# ADR-113: Tenant Provisioning and SSE Isolation

## Context
The CRM frontend relies on Server-Sent Events (SSE) to hydrate the inbox in real time without polling (FinOps optimization). The `/api/inbox/stream` endpoint connects to Postgres and listens for `inbox_updates` using a dynamically assigned search path: `SET search_path = "tenant_${safeTenantId}", public`.

## Problem
During testing, the frontend failed to connect to the SSE stream (HTTP 400). The failure occurs because the backend requires `tenant_id` to be present in the user's JWT metadata (`user_metadata`), but the current provisioning flow does not inject this value upon account creation.

## Decision
1. **JWT Metadata Injection:** All tenant users must have `{"tenant_id": "<tenant_name>"}` injected into their `raw_user_meta_data` in the Supabase Auth system at provisioning time.
2. **Strict Validation:** The `/api/inbox/stream` API will maintain strict validation. If `tenant_id` is missing, it will return HTTP 400. It is the responsibility of the provisioning flow (Mission Control / Webhook) to ensure data integrity.

## Action Plan
- **Immediate Mitigation:** Manually patch the existing test user (`fleetco@fleetco.mx`) in Supabase to include `tenant_id: "fleetco"`.
- **Systemic Fix:** Delegate to the Executor the task of refactoring the `crm-mission-control` onboarding module/webhook to automate the injection of `tenant_id` into the user metadata.
