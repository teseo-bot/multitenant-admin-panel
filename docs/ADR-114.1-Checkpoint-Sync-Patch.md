# ADR-114.1: Parche de Sincronización de Checkpoint — Mensajes Outbound del Vendedor

## Estado
**Propuesto** — Complementa al ADR-114

## Contexto: La Deuda Técnica

El ADR-114 resolvió la amnesia del bot al introducir el `PostgresCheckpointer` de LangGraph vinculado a Neon Tech. Sin embargo, existe una **brecha de sincronización crítica** que no fue contemplada: el flujo de mensajes **outbound iniciados por el vendedor humano** desde el CRM (Mission Control).

### El Flujo Actual (Roto)

Cuando el SDR humano responde a un lead desde el Inbox de Mission Control, el mensaje sigue este camino:

```
Vendedor (Frontend Mission Control)
    │
    ▼
POST /api/v1/outbound
    │
    ├──► Neon Tech (INSERT en inbox_messages, rol: 'outbound')
    │
    └──► Meta API (WhatsApp Business API — envío real al lead)
```

**El checkpoint de LangGraph nunca se actualiza.**

Resultado: cuando el lead responde al mensaje del vendedor, el Webhook del Orquestador invoca a LangGraph pasando el `thread_id` del lead. LangGraph recupera su último checkpoint y **no tiene registro de lo que el vendedor dijo**. El grafo opera con ceguera de estado parcial: ignora que un humano intervino y puede contradecir, repetir, o continuar un hilo que ya fue resuelto manualmente.

---

## Problema Formal

> **El agente LangGraph pierde la continuidad narrativa de la conversación cada vez que un vendedor humano interviene desde el Frontend. El checkpoint de Neon Tech solo refleja los mensajes procesados por el grafo, no los mensajes inyectados externamente por el agente humano.**

### Síntomas Observables
1. El SDR autónomo vuelve a preguntar información que el vendedor ya obtuvo en el hilo.
2. El bot puede ofrecer un precio o condición que el vendedor ya negoció de forma diferente.
3. El `pipeline_status` del lead puede revertirse o contradecirse si el grafo no sabe que el human-handoff ya fue parcialmente resuelto.

---

## Decisión Técnica

### Arquitectura de la Solución

El endpoint `POST /api/v1/outbound` en **Mission Control** (el CRM frontend) debe extender su responsabilidad. Después de persistir el mensaje en Neon y enviarlo por Meta, **debe notificar al Orquestador** para que éste actualice el estado del grafo LangGraph mediante su API interna.

```
Vendedor (Frontend Mission Control)
    │
    ▼
POST /api/v1/outbound   [Mission Control]
    │
    ├──► Neon Tech (INSERT en inbox_messages)
    ├──► Meta API (envío al lead)
    │
    └──► POST /api/internal/graph/inject-outbound   [Orquestador / Hono]
              │
              └──► graph.updateState(config, values)   [LangGraph]
                        │
                        └──► Neon Postgres Checkpointer (actualización del thread)
```

**Razón de la separación de responsabilidades:** Mission Control no debe tener acceso directo a la instancia del grafo LangGraph. Ese acoplamiento violaría el principio de aislamiento de contexto del Orquestador. Mission Control delega mediante HTTP al Orquestador, que es el único propietario de la instancia del grafo compilado.

---

## Mecanismo Técnico: ¿Rol `ai` o `update_state`?

### La pregunta central

¿Debemos inyectar el mensaje del vendedor como rol `'ai'` (fingiendo que el bot lo dijo) o usar la API oficial de LangGraph `graph.updateState()`?

### Análisis de Opciones

#### Opción A: Inyección directa como `AIMessage` (rol `'ai'`) — ✅ SELECCIONADA

El mensaje del vendedor se inyecta en el array de `messages` del estado de LangGraph como un `AIMessage` con metadata adicional que lo marca como intervención humana.

**Fundamento arquitectónico:**
- En el modelo conversacional de LangGraph (y LangChain), el array `messages` representa el **hilo desde la perspectiva del canal**. El Lead envía `HumanMessage`s; el agente (sea bot o humano hablando en nombre del sistema) emite `AIMessage`s.
- Al inyectar como `AIMessage`, el grafo mantiene la alternancia `Human → AI` que los modelos esperan, evitando errores como `HumanMessage consecutivos` o `ToolMessage huérfano` que ya produjeron bugs documentados en el ciclo del ADR-113.
- La metadata `human_agent_override: true` preserva la trazabilidad sin alterar la semántica del modelo.

#### Opción B: Inyección como `HumanMessage` — ❌ RECHAZADA

Inyectar el mensaje del vendedor como `HumanMessage` crearía una ambigüedad semántica irreparable: el LLM interpretaría que el **lead** dijo ese mensaje, contaminando la calificación ICP y el perfil del cliente.

#### Opción C: Nodo `update_state` con rol `'tool'` — ❌ RECHAZADA

Un `ToolMessage` requiere un `tool_call_id` correspondiente a una invocación de herramienta previa. Inyectarlo de forma libre produciría el error `ToolMessage orphan` ya documentado en el historial de la sesión de `2026-04-19`.

### Mecanismo de Actualización de Estado (LangGraph `updateState`)

LangGraph expone el método `graph.updateState(config, values, asNode?)` para mutar el estado de un thread sin ejecutar el grafo. Este es el único camino aprobado.

**Parámetros clave:**
- `config.configurable.thread_id`: El `lead_id` que identifica el hilo de conversación.
- `values.messages`: Array con el nuevo mensaje a agregar (reducer `add_messages` hará el merge).
- `asNode`: Se debe pasar `"__end__"` para indicar que este es el estado final del último ciclo, evitando que LangGraph interprete que hay un nodo pendiente de ejecución.

### Payload HTTP para el Orquestador

El endpoint `POST /api/internal/graph/inject-outbound` del Orquestador debe aceptar el siguiente payload:

```json
{
  "thread_id": "<lead_id>",
  "message": {
    "type": "ai",
    "content": "<texto_enviado_por_el_vendedor>",
    "additional_kwargs": {
      "human_agent_override": true,
      "agent_id": "<id_del_usuario_en_supabase_auth>",
      "agent_name": "<nombre_del_vendedor>",
      "channel": "whatsapp",
      "sent_at": "<ISO_8601_timestamp>"
    }
  },
  "as_node": "__end__"
}
```

**Descripción de campos:**
| Campo | Tipo | Descripción |
|---|---|---|
| `thread_id` | `string` | El `lead_id` de Neon. Es la clave del checkpoint en Postgres. |
| `message.type` | `"ai"` | Fijo. Representa al sistema respondiendo en nombre del canal. |
| `message.content` | `string` | El texto exacto enviado al lead por el vendedor. |
| `message.additional_kwargs.human_agent_override` | `boolean` | Flag de trazabilidad. Permite al grafo y a los logs distinguir mensajes de bot vs. humano. |
| `message.additional_kwargs.agent_id` | `string` | ID del usuario autenticado (Supabase Auth). Auditoría y rotación de agentes. |
| `as_node` | `"__end__"` | Fijo. Le indica a LangGraph que el checkpoint debe quedar en estado "completado". |

## Consideraciones de Consistencia

### Manejo del `pipeline_status`

Si el lead está en estado `paused` (handoff activo, per `RFC-037-HITL-Handoff-DLQ.md`), el Orquestador **no debe reactivar el grafo** al recibir este evento. La inyección de estado es silenciosa: solo actualiza el checkpoint, no desencadena ningún nodo.

El flujo de reactivación del bot permanece exclusivamente controlado por el vendedor mediante el endpoint de handoff-release.

### Idempotencia

El endpoint `inject-outbound` debe ser idempotente bajo reintentos de red. Se recomienda que Mission Control incluya un `idempotency_key` (ej. el `message_id` de Neon) en el header de la petición al Orquestador.

### Orden de Operaciones

Para garantizar la consistencia, el endpoint `POST /api/v1/outbound` en Mission Control debe ejecutar las operaciones en el siguiente orden estricto:

1. `INSERT` en `inbox_messages` de Neon Tech.
2. Envío a Meta API (WhatsApp Business).
3. `POST /api/internal/graph/inject-outbound` al Orquestador.

El paso 3 es no bloqueante para la respuesta al Frontend. Si falla, debe encolar el evento para reintento (DLQ) y NO revertir los pasos 1 y 2. La consistencia eventual es aceptable; la duplicación de mensajes al lead no lo es.

## Consecuencias

### Positivas
- El agente LangGraph recupera la visión completa del hilo conversacional, incluyendo las intervenciones humanas.
- Se elimina el riesgo de contradicción entre el bot y el vendedor en el mismo hilo.
- Se preserva la estructura de mensajes esperada por el LLM (`Human → AI → Human → AI`), evitando los bugs de secuencia documentados.

### Negativas / Trade-offs
- Se introduce un nuevo contrato HTTP interno entre Mission Control y el Orquestador. Debe ser documentado en el contrato de API interno y protegido con la misma clave de autenticación interna existente.
- El Orquestador asume una responsabilidad adicional como "guardián del estado del grafo". Esto es aceptable y coherente con su rol actual.
